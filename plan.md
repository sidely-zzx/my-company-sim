# 外包公司模拟器初步开发计划

## Summary

当前项目是一个空的 Vite + React + TypeScript 壳，依赖已包含 Zustand、Three.js、React Three Fiber 和 Electron。建议先做一个“表格经营 + 事件驱动”的可玩 MVP：玩家用初始资金招聘员工、接人力外包和项目外包，通过每日结算、员工产出、甲方满意度、违约金和工资社保支出形成核心循环。Three.js 员工办公场景和 Steam/Electron 发布放到后续阶段。

## Key Changes

- 建立游戏核心模型：`Employee`、`Resume`、`LaborOutsourceContract`、`ProjectOutsourceContract`、`ClientCompany`、`GameEvent`、`GameState`。
- 使用 Zustand 做唯一游戏状态源，先实现时间流逝、资金流水、员工列表、合同列表、招聘市场、项目市场和事件日志。
- 实现游戏时间：现实 2 秒 = 游戏 1 分钟，支持暂停、1x、2x；每日触发工资、社保、合同收入、违约检查和市场刷新。
- 首屏先做经营仪表盘，不做落地页：资金、日期、员工、合同风险、今日事件、招聘、项目、人力外包几个工作区。
- 前期 UI 以密集表格和管理面板为主，保持适合经营游戏的操作效率；后期把员工状态映射到 3D 工位和办公室表现。

## Implementation Plan

- 第一阶段：游戏内核
  - 新增 `src/game`：时间 ticker、随机生成器、结算系统、合同推进系统、招聘系统、事件系统。
  - 新增 `src/store/gameStore.ts`，集中管理游戏状态和玩家操作。
  - 固定一套初始数据：5000 初始资金、少量甲方公司、候选人模板、项目模板。

- 第二阶段：可玩 MVP
  - 招聘：每日 3 次刷新，每次 5 份简历；发 offer 时填写日薪和社保基数比例，按薪资匹配度、社保、候选人倾向计算接受概率。
  - 人力外包：区分急召和普通招聘；未按期安排员工则每日赔偿；员工产出不足时进入 1 天整改期，仍不满足则终止。
  - 项目外包：按产品、设计、开发、测试四阶段推进；员工技能和等级影响效率；截止时按完成度结算收入或违约金。
  - 员工：显示公开信息和可管理字段，隐藏能力、摸鱼倾向、仲裁倾向等仅通过事件和结果间接体现。
  - 财务：每天结算工资、社保、外包收入、违约赔偿，并写入流水和事件日志。

- 第三阶段：内容和中国互联网文化包装
  - 事件文案加入外包、甲方、面试、日报、工位、摸鱼、需求变更、验收扯皮等主题。
  - 甲方公司、项目名称、简历描述使用半随机模板生成，避免太快重复。
  - 加入风险事件：需求变更、甲方催进度、员工离职、候选人鸽 offer、满意度预警、仲裁风险提示。

- 第四阶段：3D 和 Electron
  - Three.js 只作为表现层，不承载核心结算逻辑；员工、工位、摸鱼、工作状态从 Zustand 状态映射到 3D 场景。
  - 先做办公室俯视场景、工位占用、员工状态动画，再逐步增加模型、贴图和交互。
  - Electron 后期补齐主进程、存档路径、窗口配置、构建配置和 Steam 发布所需资源。

## Public Interfaces And Types

- `GameState`：包含时间、速度、资金、员工、简历池、招聘公告、合同、项目、事件、财务流水。
- `Employee`：公开字段包括姓名/花名、学校、简历技能、岗位、日薪、社保比例、当前分配；隐藏字段包括真实能力、摸鱼倾向、满意度、仲裁倾向、年限。
- `ProjectOutsourceContract`：金额、违约金、截止时间、阶段、人力需求、完成度、已分配员工、状态。
- `LaborOutsourceContract`：甲方预算、岗位要求、等级要求、招聘期限、员工绑定、满意度、整改截止、状态。
- `GameAction`：招聘、发 offer、接受合同、分配员工、解雇员工、调整薪资社保、切换速度、刷新市场。

## Test Plan

- 单元测试游戏纯逻辑：时间推进、每日结算、项目阶段推进、合同违约、人力满意度、offer 接受概率边界。
- 状态测试：连续推进多天后资金、合同状态、员工状态和事件日志一致。
- UI 手测场景：开局招聘、接急召人力、接项目外包、员工产出不足、项目逾期、解雇赔偿、资金归零。
- 构建检查：每个阶段保持 `pnpm build` 和 `pnpm lint` 可通过。

## Assumptions

- MVP 优先做单机离线经营，不做账号、联网、云存档和多人。
- 前期不追求真实劳动法完整模拟，只保留“社保基数、辞退赔偿、仲裁风险”作为核心玩法变量。
- 所有随机内容先用可复现的模板和权重生成，后期再扩展内容库。
- Three.js 和 Electron 不进入第一版核心玩法实现，等表格版循环好玩后再接入。
